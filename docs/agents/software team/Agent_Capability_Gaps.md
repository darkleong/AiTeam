# Agent 能力缺口清單

> 版本：v1.1
> 建立日期：2026-04-03
> 最後更新：2026-04-03
> 說明：基於程式碼實際分析，記錄每個 Agent 目前的能力邊界與需要補強的方向。
> 目的：為「Agent 內部協作機制」奠定基礎，讓系統可以在不需老闆逐一干預的情況下自行運轉。

---

## 架構決策：CEO 作為協作中樞

> **決策日期：2026-04-03**

### 不採用「Agent 直接溝通」的原因

如果讓 Agent 彼此直接通訊，會形成複雜的網狀結構：

```
Dev ←→ Rosa
Dev ←→ Demi
Vera → Dev
Maya → Dev → CEO
Rosa ←→ Demi
...
```

每個 Agent 都要知道其他 Agent 的存在，新增 Agent 時，每個現有 Agent 都需要跟著改。

### 採用「CEO 作為中樞」的架構

```
所有 Agent 只認識一個對象：CEO

Dev 遇到問題   → CEO → 判斷找誰 → Rosa / Demi / 老闆
Vera 發現 🔴   → CEO → 判斷下一步 → 通知 Dev 修 / 升級老闆
Maya 偵測失敗  → CEO → 判斷下一步 → 觸發回滾 / 通知 Dev
```

每個 Agent 只需要一個溝通管道。這個架構：
- **符合現有設計**：CEO 已是路由中心，只需延伸成「執行中也能路由」
- **完整可觀測**：所有溝通都過 CEO，老闆在 Discord 看得到全程
- **自然升級路徑**：Stage 9 CEO 升級的方向，正是讓 CEO 從「任務路由器」升級為「任務生命週期全程指揮官」

### CEO 需要擴充的能力

| 能力 | 說明 |
|------|------|
| 接收 Agent 提問 | `waiting_input` 回傳格式：Agent 暫停並附上問題與上下文 |
| 任務狀態擴充 | `pending → running → waiting_input → running → done / failed` |
| 問題路由判斷 | 需求細節 → Rosa；UI 規格 → Demi；商業決策 → 老闆 |
| 自動觸發下一步 | 任務完成後 CEO 主動決定下一個 Agent，不等老闆指示 |

### CEO 路由輕量化原則

> **決策日期：2026-04-03**
> Agent 回報「完成」時，CEO 不需要呼叫 LLM，按標準流程往下即可。只有在真正需要判斷的情況才呼叫 LLM。

| Agent 回報內容 | CEO 處理方式 | 是否需要 LLM |
|--------------|------------|------------|
| 完成（done） | 查任務流程表，直接觸發下一步 | ❌ 不需要 |
| 失敗（failed）+ 已知原因 | 按失敗類型對應處理（外部故障重試、程式問題通知老闆） | ❌ 不需要 |
| 等待輸入（waiting_input）+ 問題類型明確 | 按問題類型路由（需求 → Rosa，UI → Demi，商業決策 → 老闆） | ❌ 不需要 |
| 等待輸入 + 問題類型不明確 | 呼叫 LLM 判斷要找誰 | ✅ 需要 |
| 老闆下新指令 | 呼叫 LLM 解析意圖、分類、決定派工 | ✅ 需要 |

**好處：** 大多數 Agent 回報走快速路徑，不卡在 LLM 呼叫上，CEO 不會成為系統瓶頸。

## 核心問題：各 Agent 的輸入來源太窄，彼此孤立

目前每個 Agent 收到任務時，只能看到 CEO 傳入的「任務標題 + 描述」，
看不到其他 Agent 的工作成果，也無法主動查詢 codebase 或提出疑問。

## CEO（Victoria）

### 現有能力
- 讀取 DB 近期任務（5 筆）作為上下文
- 多輪對話歷史（最多 6 輪）
- 接收 Discord 截圖
- 不確定時反問（每次一個問題）

### 缺口

| 缺口 | 說明 | 已排入 |
|------|------|--------|
| 分類判斷能力 | 目前直接派工，無法主動判斷 Bug / 新功能 / 正常行為 / 疑問 | Stage 9 |
| 分類前查資料 | 分類前應主動查 GitHub Issues / PR / DB 任務記錄 | Stage 9 |
| 提案模式 | 新功能時協調 Rosa + Demi 並行產出，彙整提案書 | Stage 9 |
| 接收 Agent 提問並路由 | 任務執行中途，Agent 暫停並回報問題，CEO 判斷找誰回答後繼續 | 未排入 |
| 任務暫停/恢復機制 | 任務狀態需支援 `waiting_input`，恢復時帶著答案重新喚醒 Agent | 未排入 |
| 任務完成後自動觸發下一步 | Vera 審查完有 🔴 → 自動叫 Dev 修；Dev 修完 → 自動叫 Vera 重審 | 未排入 |

## Dev（Cody）

### 現有能力
- 接收任務標題 + 描述
- 讀取指定檔案的現有程式碼（逐檔）
- Git 操作：Clone → Branch → 改寫檔案 → Commit → Push → 開 PR
- Code Review 模式（列出所有檔案，LLM 審查）

### 缺口

| 缺口 | 說明 | 優先級 |
|------|------|--------|
| **看不到 codebase 結構** | 制定計畫時（BuildPlanAsync）只有任務標題 + repo 名，需要猜 `files_to_modify`，高機率猜錯 | 🔴 高 |
| **看不到 Rosa 的 Issues** | 沒有明確的驗收條件，不知道做完後要達到什麼目標 | 🔴 高 |
| **看不到 Demi 的 UI 規格** | UI 實作完全靠 LLM 自行發揮，可能跟設計文件不一致 | 🔴 高 |
| 無法暫停提問 | 遇到規格不清楚時只能猜或直接 failed，無法向 CEO 請求釐清 | 🟡 中 |
| 逐檔改寫缺乏上下文 | 每個檔案獨立呼叫 LLM，不知道其他檔案的改動，可能產生不一致的程式碼 | 🟡 中 |

### 理想流程（補充上下文後）

```
CEO 派任務給 Dev，同時附上：
  - Rosa 建立的 GitHub Issues（含驗收條件）
  - Demi 產出的 UI 規格文件路徑
  - Repo 的目錄結構快照
    ↓
Dev 讀取後制定計畫，確認 files_to_modify
    ↓
Dev 遇到不確定（如：Issues 描述有歧義）
  → 向 CEO 回報 waiting_input + 問題
  → CEO 找 Rosa 補充 → 答案回傳給 Dev
    ↓
Dev 繼續實作 → 開 PR
```

|------|------|--------|

| 無法驗證功能正確性 | 只測「程式碼有沒有壞掉」，不測「功能有沒有做對」 | 🔴 高 |

## Doc（Sage）

### 現有能力
- 讀取指定路徑下的所有 .cs 檔案
- 兩種模式：Markdown API 文件 / XML `<summary>` 註解補充
- 開 PR

### 定位釐清

> **Sage 的文件不是只給人看的，應該是給整個 Agent 團隊看的。**
>
> 現在：Sage 產出 Markdown 文件，沒有其他 Agent 會去讀它。
> 理想：Sage 建立的是「共用知識庫」，Dev 制定計畫前讀它、Rosa 分析需求前讀它、Vera 審查時對照它。
> 換句話說：Sage 是在把 codebase 轉換成「Agent 可以理解的知識」，是解決「各 Agent 看不到現有實作」這個根本問題的關鍵角色。

### 缺口

| 缺口 | 說明 | 優先級 |
|------|------|--------|
| 文件定位不明，沒有 Agent 會主動讀 | 需要建立「Agent 查詢文件」的機制，讓文件真正有用 | 🔴 高（但依賴 CEO 中樞機制先完成）|
| 文件缺少需求背景 | 只能描述「程式碼做了什麼」，不知道「為什麼這樣設計」 | 🟡 中 |
| 沒有讀 PR 描述 | 沒有讀取 PR 的說明內容，文件缺少變更目的的上下文 | 🟡 中 |
| 不知道已有哪些文件 | 可能產出與現有文件重複的內容 | 🔵 低 |

## Reviewer（Vera）

### 現有能力
- 讀取 PR 的 .cs 檔案 + diff patch + 完整內容
- 五面向審查：邏輯/品質/效能/安全/可維護性
- 🔴/🟡/🟢 分級，發 GitHub Review 評論

### 缺口

| 缺口 | 說明 | 優先級 |
|------|------|--------|
| **看不到需求，只審程式碼品質** | 能判斷「程式碼寫得好不好」，但無法判斷「功能有沒有做對」 | 🔴 高 |
| 不知道驗收條件 | Rosa 定義的 acceptance criteria 不會自動傳給 Vera | 🔴 高 |
| 審查完沒有後續動作 | 發完評論就停了，需要人工傳達給 Dev；🔴 問題修完後也無法自動重審 | 🔴 高 |
| 不知道 UI 規格 | 無法審查 Blazor 元件是否符合 Demi 的設計規格 | 🟡 中 |

### 理想流程

```
Vera 審查 PR，發現 🔴 critical issues
    ↓
Vera 在 GitHub 留評論（現有行為）
    ↓
Vera 向 CEO 回報：「PR #X 有 2 個必修問題」
    ↓
CEO 判斷 → 通知 Dev：「Vera 要求修正，修完再告知我」
    ↓
Dev 修好，推新 commit
    ↓
CEO 偵測到 PR 有新 commit → 自動再派 Vera 重審
    ↓
Vera 確認無 🔴 → 回報 CEO
    ↓
CEO 通知老闆：「PR #X 已通過審查，可以 merge」
```

## Ops（Maya）

### 現有能力
- GitHub Actions 監控（排程）：外部故障自動重試，程式問題通知老闆
- 部署監控：DB ping 確認服務狀態
- 健康檢查（排程）：記憶體 + DB 連線

### 缺口

| 缺口 | 說明 | 優先級 |
|------|------|--------|
| **無法自動回滾** | Bot 容器內無法執行 docker-compose，失敗只能通知老闆手動處理 | 🔴 高 |
| CI/CD 失敗分類不精確 | 目前靠 job 名稱關鍵字（"push image"）判斷外部/程式故障，可能誤判 | 🟡 中 |
| 健康檢查範圍有限 | 只檢查記憶體和 DB，無法檢查 Discord Bot 是否正常連線 | 🔵 低 |

### 理想流程

```
CI/CD 失敗，Maya 判斷為程式問題
    ↓
Maya 執行 rollback（需要：外部 rollback workflow 或 docker API）
    ↓
Maya 向 CEO 回報：「v1.2.0 部署失敗，已回滾到 v1.1.0，原因：Build 錯誤」
    ↓
CEO 判斷 → 通知老闆（現況）+ 通知 Dev（「請修復後重新提交」）
    ↓
Dev 修復 → 推新 commit → 觸發新 CI/CD
    ↓
Maya 繼續監控
```

**Rollback 技術方向：**
Bot 容器內無法呼叫 docker-compose。解法：在 self-hosted runner 上建立一個「rollback」GitHub Actions workflow，Maya 透過 GitHub API 觸發它執行回滾。

## 最優先解決的問題

讓整個開發流程「閉環」的關鍵，按依賴順序排列：

| 順序 | 問題 | 影響 Agent | 備註 |
|------|------|-----------|------|
| 1 | CEO 擴充：接收 Agent 回報、路由問題、自動觸發下一步 | 全部 | 其他一切的基礎 |
| 2 | Dev 制定計畫前能掃描 codebase 結構 | Dev | 解決猜錯檔案的問題 |
| 3 | 上游產出（Issues / UI 規格）由 CEO 自動附帶傳給下游 | Dev、QA、Vera | Rosa/Demi 的工作才有意義 |
| 4 | Vera 審查後自動通知 Dev，Dev 修完後自動重審 | Vera、Dev | 讓 Review 流程閉環 |
| 5 | Sage 的文件成為 Agent 可查詢的知識庫 | 全部 | 解決「看不到現有實作」的根本問題 |
| 6 | Maya 能透過 GitHub API 觸發 rollback workflow | Ops | 解決部署失敗只能通知人的問題 |
