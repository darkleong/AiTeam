# UI 規格文件：Reviewer 報告問題總結統計行

---

## 確認任務範圍

在輸出規格前，我需要先說明範圍判斷：

此需求的**核心變更發生在後端 Reviewer Agent（Vera）的報告產出邏輯**，屬於文字處理與格式附加，並非獨立頁面或新 UI 元件的建立。因此本文件的規格重心在於：

- 報告輸出內容的格式規格（統計行的文字結構）
- 報告顯示元件的呈現規格（前端如何渲染這一行）
- Streaming 輸出時的特殊考量

如果此任務**不包含前端顯示頁面的新建**，僅涉及報告文字格式與現有顯示元件的相容性，請告知，我可以聚焦在格式規格與注意事項即可。以下規格以「現有報告顯示頁面已存在，本次只新增統計行」為前提撰寫。

---

## 1. 頁面目的

讓 Code Review 報告的閱讀者在報告末尾能快速掌握問題嚴重度的數量摘要，無需自行逐行計算嚴重問題與建議的總數。

---

## 2. 頁面結構

本次變更不新增獨立頁面，而是在**現有報告顯示區塊的最末尾**新增一條統計摘要行。

報告顯示區塊的垂直結構由上至下為：

```
[ 報告標題 / Header ]
[ 報告本文（逐條問題列表，含 🔴 / 🟡 標記）]
[ 分隔線 ]                        ← 新增
[ 問題總結統計行 ]                 ← 新增
```

統計行為**單行純文字**，不另起區塊，緊接於分隔線之後，作為報告的最終結語。

---

## 3. 元件規格

### 3-1. 報告本文顯示區（現有元件，確認相容性）

| 項目 | 規格 |
|------|------|
| 元件 | 現有報告 Markdown 渲染元件（或 `MudText` 純文字區塊） |
| 重要行為 | 必須能正確渲染 Unicode emoji（🔴、🟡），確認現有渲染方式不會 escape 掉 emoji 字元 |
| Streaming 模式 | 若報告為 Streaming 逐字輸出，統計行**必須等全文完成後才附加**，不得在 Streaming 過程中提前顯示不完整的統計數字 |

---

### 3-2. 分隔線（新增）

| 項目 | 規格 |
|------|------|
| 元件 | `MudDivider` |
| Props | `Class="my-2"`（上下保留適當間距即可） |
| 出現時機 | 與統計行同步出現，僅在報告**完整產出後**才顯示 |

---

### 3-3. 問題總結統計行（新增，核心元件）

| 項目 | 規格 |
|------|------|
| 元件 | `MudText` |
| Props | `Typo="Typo.body1"`、`GutterBottom="false"` |
| 顯示文字格式 | `共發現 N 個 🔴 嚴重問題、M 個 🟡 建議` |
| 文字來源 | 由後端 Vera Agent 在報告本文產出完畢後計算，附加於報告字串末尾，前端直接渲染，不在前端做二次解析 |
| 出現時機 | 僅在報告**完整輸出完成後**才顯示（見互動行為第 4 點） |

**文字格式詳細規格：**

```
共發現 {N} 個 🔴 嚴重問題、{M} 個 🟡 建議
```

- `{N}`：報告本文中 🔴 出現的次數（整數，最小值為 0）
- `{M}`：報告本文中 🟡 出現的次數（整數，最小值為 0）
- 標點符號使用全形頓號「、」
- 若 N = 0 且 M = 0，仍顯示該行，文字為：`共發現 0 個 🔴 嚴重問題、0 個 🟡 建議`（不隱藏，確保報告完整性）

---

## 4. 資料來源

| 資料項目 | 來源 | 說明 |
|----------|------|------|
| 🔴 計數（N） | **後端 Vera Agent** | 在報告本文字串完整產出後，以字元/字串比對方式統計 `🔴` 的出現次數 |
| 🟡 計數（M） | **後端 Vera Agent** | 同上，統計 `🟡` 的出現次數 |
| 統計行文字 | **後端組合後附加** | 後端直接將統計行文字拼接至報告本文末尾，作為報告字串的一部分回傳，前端無需額外處理 |

> ⚠️ **重要原則**：統計邏輯集中在後端 Vera Agent，前端不做任何 emoji 計數或字串解析，只負責渲染完整報告字串。

---

## 5. 互動行為

### 5-1. 正常報告產出流程

1. 使用者觸發 Code Review（現有流程不變）
2. Vera Agent 開始產出報告本文（現有流程不變）
3. 若為 **Streaming 模式**：前端持續顯示逐字輸出的報告本文，此階段統計行與分隔線**不顯示**
4. 後端偵測到報告本文輸出完成後：
   - 統計報告本文中 🔴 與 🟡 的數量
   - 組合統計行文字
   - 在報告末尾附加分隔符號（建議使用 `\n---\n`）與統計行文字
5. 前端接收完整報告後，渲染分隔線（`MudDivider`）與統計行（`MudText`）

### 5-2. 統計行的顯示狀態

| 狀態 | 統計行顯示 |
|------|-----------|
| 報告產出中（Streaming） | **不顯示** |
| 報告產出完成 | **顯示**，文字固定不再變動 |
| 報告產出失敗 / 錯誤中斷 | **不顯示**（因無法確保計數正確） |

### 5-3. 報告複製 / 匯出行為（若現有功能支援）

若現有頁面提供「複製報告」或「匯出」功能，統計行應**包含在複製/匯出內容中**，分隔線在純文字匯出時以 `---` 表示。

## 6. 注意事項

### 6-1. Blazor Server 電路考量

- 統計行的顯示觸發屬於**一次性狀態更新**（報告完成 → 顯示），不涉及持續訂閱，無電路洩漏風險
- 若使用 `StateHasChanged()` 驅動 UI 更新，確保在報告完整接收後才呼叫，避免 Streaming 過程中頻繁觸發渲染

### 6-2. Emoji 計數的實作邊界

- 🔴（U+1F534）與 🟡（U+1F7E1）為多 byte Unicode 字元，後端以字串比對計數時，需使用 **`StringComparison.Ordinal`** 或直接以 emoji 字元字面量比對，避免因編碼問題導致計數錯誤
- 若報告本文中有引用他人程式碼的區塊（如 code block），其中的 emoji 也會被計入。若需排除，需定義明確的「計數範圍規則」並在本次確認（建議：**本次不排除 code block，全文計數，保持實作簡單**）

### 6-3. Streaming vs. 非 Streaming 模式相容

- 若 Vera Agent 支援兩種輸出模式，統計附加邏輯必須在**兩種模式下皆有效**
- Streaming 模式下，統計行應作為最後一個 chunk 附加，而非拆散於多個 chunk 中

### 6-4. 報告字串的分隔符設計

- 後端附加統計行時建議以 `\n\n---\n\n` 作為與本文的分隔，確保 Markdown 渲染時分隔線能正確呈現
- 前端若以 Markdown 渲染（而非純文字），統計行結尾不需要額外 `\n`

### 6-5. 不在本次範圍內的事項

以下項目**不在本次規格範圍**，如有需要請另開任務：

- 統計行的樣式客製化（顏色強調、粗體等）
- 依嚴重程度排序或分類問題
- 統計數字的點擊跳轉（錨點定位到對應問題）
- 歷史報告的統計行補充（舊報告不補算）

1. 現有報告顯示元件是以 **Markdown 渲染**還是**純文字渲染**？這影響分隔線的格式選擇
2. Vera Agent 目前是否為 Streaming 輸出？或是一次性回傳完整報告字串？
3. 🔴 / 🟡 是否只出現在特定格式的列表行首（如 `- 🔴 ...`），或可能散落在內文任意位置？這影響計數邏輯的精確度設計
