# UI 規格文件：Reviewer 報告總結統計行

> **文件版本**：v1.0  
> **對應功能**：Reviewer Agent（Vera）Code Review 報告統計摘要  
> **目標讀者**：Dev Agent（開發工程師）

---

## 確認任務範圍

在輸出規格前，先確認我對此需求的理解：

- **作用範圍**：Vera 產出報告的**最後一行**，純文字附加，不是獨立頁面
- **觸發時機**：報告**產出時**自動計算，非使用者手動操作
- **計算對象**：掃描報告內容中 🔴（嚴重）與 🟡（建議）的標記數量
- **顯示位置**：現有報告展示頁面或元件的末尾區塊

以下規格基於此理解產出。若有偏差請提出修正。

---

## 1. 頁面目的

在 Vera 產出的 Code Review 報告末尾，自動附加一行可讀性高的統計摘要，讓 Reviewer 與開發者能快速掌握本次審查的問題分佈，無需人工逐行計算。

---

## 2. 頁面結構

此功能**不新增獨立頁面**，而是在現有「Code Review 報告展示區塊」的末尾插入一個統計摘要列。

**報告展示區塊的整體結構（由上至下）：**

```
┌─────────────────────────────────────────┐
│  報告標題列（PR 名稱、時間、Reviewer）      │
├─────────────────────────────────────────┤
│                                         │
│  報告主體內容                             │
│  （各條 🔴 嚴重問題 / 🟡 建議 條目）       │
│                                         │
├─────────────────────────────────────────┤
│  ── 統計摘要列（新增區塊）──               │
│  共發現 N 個 🔴 嚴重問題、M 個 🟡 建議     │
└─────────────────────────────────────────┘
```

統計摘要列與報告主體之間以**視覺分隔線**區隔，強調此行為系統自動產生的摘要，而非報告正文的一部分。

---

## 3. 元件規格

### 3-1. 統計摘要列容器

| 屬性 | 規格 |
|------|------|
| **元件** | `MudDivider` + `MudText` 組合 |
| **排版** | `MudDivider` 在上方作為分隔，`MudText` 緊接其後 |
| **位置** | 報告內容區塊的最末尾，固定於報告底部 |

### 3-2. `MudDivider`（分隔線）

| Props | 值 |
|-------|----|
| `DividerType` | `DividerType.FullWidth` |
| `Class` | 保留適當的上下間距（使用 MudBlazor spacing class） |

> **用途**：視覺上區隔報告正文與系統產生的統計摘要

### 3-3. `MudText`（統計摘要文字）

| Props | 值 |
|-------|----|
| **元件** | `MudText` |
| `Typo` | `Typo.body2` |
| `Align` | `Align.Start`（靠左對齊，與報告正文一致） |
| **顯示文字格式** | `共發現 {criticalCount} 個 🔴 嚴重問題、{suggestionCount} 個 🟡 建議` |

**顯示範例：**
```
共發現 3 個 🔴 嚴重問題、7 個 🟡 建議
共發現 0 個 🔴 嚴重問題、2 個 🟡 建議
```

> **注意**：當兩者皆為 0 時，仍須顯示此行（`共發現 0 個 🔴 嚴重問題、0 個 🟡 建議`），不得隱藏，避免使用者誤以為報告未產出。

## 4. 資料來源

### 4-1. 統計數字的計算來源

| 資料項目 | 來源 | 說明 |
|----------|------|------|
| `criticalCount`（🔴 嚴重問題數） | **報告內容字串解析** | 掃描 Vera 產出的報告文字，計算 🔴 emoji 出現次數 |
| `suggestionCount`（🟡 建議數） | **報告內容字串解析** | 掃描 Vera 產出的報告文字，計算 🟡 emoji 出現次數 |

### 4-2. 計算時機

- **觸發點**：Vera Agent 完成報告產出、報告文字寫入完成後
- **計算方式**：由後端 Service 層負責解析報告字串，回傳統計結果；**前端不做字串解析邏輯**
- **資料傳遞**：統計結果隨報告資料一併傳入前端元件（建議作為 ViewModel 的屬性附加）

### 4-3. 建議 ViewModel 結構（供 Dev 參考）

> 以下為結構描述，非程式碼規格：

報告 ViewModel 應包含以下欄位：
- 原有報告內容欄位（維持不變）
- 新增 `CriticalCount`（整數，🔴 嚴重問題數量）
- 新增 `SuggestionCount`（整數，🟡 建議數量）
- 新增 `SummaryLine`（字串，完整統計文字，由後端組合）—— *可選，若前端直接組合可省略*

## 5. 互動行為

此功能以**自動呈現**為主，無使用者主動觸發的互動。以下為相關狀態說明：

### 5-1. 報告載入中

| 狀態 | 統計摘要列行為 |
|------|--------------|
| 報告尚未產出 / 載入中 | 統計摘要列**不顯示**（整個區塊隱藏） |
| 報告產出完成 | 統計摘要列**自動顯示** |

### 5-2. 報告串流輸出（若 Vera 採用逐字輸出模式）

若 Vera 的報告採用 **串流（Streaming）** 方式逐字輸出：

- 統計摘要列應等到**串流完全結束**後才顯示，不得在輸出中途顯示（避免數字隨輸出變動造成跳動）
- 串流結束事件由後端通知前端（透過 SignalR），前端接收到「報告完成」事件後才渲染統計行

### 5-3. 報告重新產出

若使用者觸發重新 Review（重跑 Vera）：

- 統計摘要列應隨報告內容一併清除並重置
- 新報告產出完成後，重新顯示新的統計結果

## 6. 注意事項

### 6-1. Blazor Server 電路隔離

- 統計數字的**計算邏輯不得放在 UI 元件內**，應集中在 Service 層，避免各使用者電路之間的資料污染
- 若多個使用者同時觀看不同報告，各電路的統計數字必須獨立，透過 component-level 的 `@inject` 取得 Scoped Service 確保隔離

### 6-2. Emoji 計算的穩定性

- 🔴 與 🟡 emoji 在不同作業系統、字型渲染下為多 byte 字元，計算時應以 **Unicode 碼位**為單位計算，而非位元組長度
- 若 Vera 的報告格式未來有調整（如改用文字標記取代 emoji），計算邏輯應設計為**可替換的策略**，建議以 Interface 或設定值抽象化標記符號

### 6-3. 效能考量

- 報告內容可能較長（數百行），字串掃描應在**後端一次性計算**，結果快取至 ViewModel，避免每次 Blazor re-render 時重複計算
- 禁止在 `OnAfterRenderAsync` 或 UI 生命週期中執行字串解析

### 6-4. 空報告邊界情境

| 情境 | 預期行為 |
|------|----------|
| 報告內容為空字串 | 顯示 `共發現 0 個 🔴 嚴重問題、0 個 🟡 建議` |
| 報告內容為 null | 統計摘要列整體不渲染，對應上方「載入中」狀態 |
| 僅有嚴重問題，無建議 | 正常顯示，`🟡 建議` 數字顯示 0 |
