# UI 規格文件：Reviewer Agent 報告統計總結

---

## 任務範圍確認

在開始輸出規格前，我需要先說明此任務的範圍判斷：

這個需求的核心是「**後端報告生成邏輯**」的修改（解析報告內容、計算分級數量、附加總結行），**並非 UI 頁面功能**。但報告的呈現載體（顯示報告的頁面/元件）仍屬 UI 規格範疇，因此本文件將涵蓋：

- ✅ 報告呈現區塊的 UI 規格（統計總結行的顯示方式）
- ✅ 報告內容格式的顯示規格（供 Dev Agent 實作解析邏輯時對齊）
- ❌ 不涵蓋後端解析演算法的實作細節（屬 Backend Agent 工作）

如果需求範圍有誤，請在繼續前確認。

---

## 1. 頁面目的

在 Reviewer Agent（Vera）產出的 Code Review 報告尾端，自動附加一行統計摘要，讓使用者能快速掌握本次審查中嚴重問題與建議的數量總覽，無需逐行閱讀完整報告。

---

## 2. 頁面結構

本功能作用於「**Code Review 報告顯示區塊**」，不新增獨立頁面，而是修改現有報告輸出元件的尾端內容。

頁面結構描述如下：

```
┌─────────────────────────────────────────────┐
│  Code Review 報告區塊                         │
│                                              │
│  [報告正文]                                   │
│  - 各分級審查項目（🔴 嚴重 / 🟡 建議 等）     │
│  - ...（原有報告內容，不變動）                 │
│                                              │
│  ─────────────────────────────────────────  │
│  [統計總結行]  ← 本次新增區塊                 │
│  共發現 N 個 🔴 嚴重問題、M 個 🟡 建議        │
└─────────────────────────────────────────────┘
```

**排版說明：**
- 統計總結行位於報告正文**最末尾**，以視覺分隔線（`---` 水平線）與上方報告正文區隔
- 統計總結行為單行純文字，不換行、不分欄
- 分隔線與統計總結行同屬報告內容的一部分，隨報告一同渲染，不是獨立浮動元素

---

## 3. 元件規格

### 3.1 報告內容顯示元件（現有元件修改）

| 項目 | 規格 |
|------|------|
| **元件** | 現有報告顯示元件（推測為含 Markdown 渲染的文字區塊，請對齊現有實作） |
| **修改方式** | 在傳入的報告字串尾端，由後端或 ViewModel 層附加統計總結段落後，再交由元件渲染 |
| **渲染方式** | 統計總結行須與報告正文使用**相同的 Markdown 渲染管道**，確保 Emoji 與文字正確顯示 |

### 3.2 統計總結行（新增內容片段）

此區塊為純文字內容，附加於報告 Markdown 字串尾端，格式固定如下：

**Markdown 原始格式（後端輸出）：**
```
---
共發現 N 個 🔴 嚴重問題、M 個 🟡 建議
```

**顯示規格說明：**

| 項目 | 規格 |
|------|------|
| **分隔線** | Markdown `---`，渲染為水平線，用於在視覺上隔開正文與總結 |
| **統計文字** | 純文字行，包含 Unicode Emoji（`🔴`、`🟡`），不加粗、不斜體 |
| **N 值** | 報告中標記為 🔴（嚴重）級別的項目計數，最小值為 `0` |
| **M 值** | 報告中標記為 🟡（建議）級別的項目計數，最小值為 `0` |
| **零值顯示** | N 或 M 為 `0` 時，**仍正常顯示**，例如「共發現 0 個 🔴 嚴重問題、3 個 🟡 建議」，不隱藏 |
| **位置** | 報告字串的絕對末尾，統計行之後不再有其他內容 |

## 4. 資料來源

| 資料項目 | 來源 | 說明 |
|----------|------|------|
| 報告正文 | Reviewer Agent（Vera）產出的 Markdown 字串 | 現有機制，不變動 |
| 🔴 嚴重問題數量（N） | **後端解析邏輯**，對報告正文進行分析後計算 | 解析規則見第 5 節 |
| 🟡 建議數量（M） | **後端解析邏輯**，對報告正文進行分析後計算 | 解析規則見第 5 節 |
| 統計總結行文字 | 由後端組合 N、M 後，附加至報告字串尾端 | UI 層僅負責渲染，不計算 |

> ⚠️ **重要原則**：UI 層不負責解析或計算 N、M 數值。所有計算邏輯由後端完成，UI 僅渲染最終字串。

## 5. 互動行為

### 5.1 報告生成完成後的顯示行為

| 觸發條件 | 系統行為 |
|----------|----------|
| Reviewer Agent 完成報告生成 | 報告字串（含末尾統計總結）整體傳至 UI 層渲染 |
| 報告以串流（streaming）方式輸出 | 統計總結行**等待報告完全生成後**才附加，不在串流過程中提前顯示 |
| 報告為空或生成失敗 | 不顯示統計總結行，依現有錯誤處理邏輯呈現錯誤訊息 |

### 5.2 分級識別規則（供後端 Dev Agent 對齊 UI 預期）

UI 層預期後端依照以下規則識別分級項目，**以確保計數與報告正文視覺呈現一致**：

| 級別 | 識別標記 | 計入統計欄位 |
|------|----------|--------------|
| 嚴重問題 | 行內含 `🔴` | N（嚴重問題數） |
| 建議 | 行內含 `🟡` | M（建議數） |
| 其他級別 | 例如 `🟢`、`🔵` 等 | **不計入**此統計行 |

> 📌 若專案中存在其他分級定義（如 `🟠` 警告），請 Dev Agent 在實作前向 Christ 確認是否需擴充統計欄位，本規格目前僅涵蓋 🔴 與 🟡 兩個級別。

### 5.3 使用者操作（無新增互動）

統計總結行為**唯讀展示**，不提供點擊、篩選、展開等互動行為。使用者無法對此行進行任何操作。

## 6. 注意事項

### 6.1 Blazor Server 特殊考量

| 項目 | 說明 |
|------|------|
| **串流輸出與統計行時序** | 若報告採用 SignalR 串流逐字輸出，統計總結行必須在後端確認報告完整生成後，作為最後一個訊息片段推送，避免統計數值在串流中途出現後又被覆蓋 |
| **電路隔離** | 各使用者的報告內容與統計數值儲存於各自的 Blazor Server Circuit 中，不共享狀態，現有架構應已涵蓋此點，確認無跨 Circuit 共用報告字串的情況 |
| **StateHasChanged 呼叫** | 若報告內容以非同步方式更新（串流或 await 後賦值），確保在附加統計總結後呼叫 `StateHasChanged()` 以觸發 UI 重新渲染 |

### 6.2 效能注意

| 項目 | 說明 |
|------|------|
| **解析成本** | 統計解析在後端執行（對報告字串掃描一次），時間複雜度 O(n)，不影響 UI 渲染效能 |
| **字串拼接** | 附加統計總結行時，避免在報告字串已很長的情況下重複拼接，建議使用 `StringBuilder` 或直接在最終輸出時一次組合 |
| **Markdown 渲染** | 統計總結行加入後，整份報告字串長度增加極小（約 30 字元），對 Markdown 渲染效能無顯著影響 |

### 6.3 Emoji 顯示相容性

| 項目 | 說明 |
|------|------|
| **Unicode Emoji** | `🔴`（U+1F534）與 `🟡`（U+1F7E1）為標準 Unicode，現代瀏覽器支援正常，無需特殊處理 |
| **字型回退** | Emoji 顯示依賴系統字型，Blazor Server 環境下由瀏覽器端渲染，與後端無關，應可正常顯示 |
