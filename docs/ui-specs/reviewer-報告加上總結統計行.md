# UI 規格文件：Reviewer 報告總結統計行

> **文件版本**：v1.0  
> **對應 Agent**：Vera（Reviewer Agent）  
> **文件用途**：供 Dev Agent 實作報告生成邏輯與展示規格

---

## 確認任務範圍

在開始輸出規格之前，先釐清本次任務邊界：

本需求的核心是**報告內容生成邏輯**（統計 + 附加總結行），而非全新頁面。因此本規格文件涵蓋：

- ✅ 總結統計行的格式規格
- ✅ 統計邏輯的計算規則
- ✅ 報告展示區塊中總結行的呈現規格
- ❌ 不涉及報告列表頁、PR 選擇流程、Vera 其他輸出行為

---

## 1. 頁面目的

在 Vera 產出的 Code Review Markdown 報告末尾，自動附加一行總結統計，讓閱讀者在不逐行掃描報告的情況下，快速掌握本次 Review 的問題嚴重程度分布。

---

## 2. 頁面結構

本功能影響的 UI 區塊為「報告展示區（Report Viewer）」，結構描述如下：

**報告展示區整體佈局（由上至下）：**

1. **報告標題列**：顯示 PR 編號、Review 時間戳、Reviewer Agent 名稱（Vera）
2. **報告主體區（Markdown 渲染區）**：逐條顯示 Review 項目，包含嚴重問題與建議
3. **總結統計行（新增區塊）**：位於報告主體末尾，視覺上與主體內容同屬一個容器，但以分隔線與上方內容區隔，呈現統計摘要文字

> 總結統計行不是獨立的 UI 元件區塊，而是 Markdown 報告內容的一部分，附加於報告字串末尾後一併渲染。

---

## 3. 元件規格

### 3-1. 報告展示容器

| 項目 | 規格 |
|------|------|
| 元件 | `MudPaper` |
| 用途 | 包覆整個報告展示區，提供視覺邊界 |
| 重要 Props | `Elevation="2"`、`Class="pa-4"` |
| 行為 | 靜態展示，無互動 |

---

### 3-2. Markdown 渲染區（含總結統計行）

| 項目 | 規格 |
|------|------|
| 元件 | 現有 Markdown 渲染元件（專案現行使用者，保持一致） |
| 用途 | 將 Vera 產出的報告字串（含總結行）渲染為格式化內容 |
| 資料來源 | 綁定報告內容字串（已含總結統計行） |
| 行為 | 總結行為 Markdown 字串的一部分，隨主體一同渲染，無需額外元件處理 |

> **注意**：總結統計行以純 Markdown 格式寫入報告字串，不需要獨立的 Blazor 元件。渲染端無需改動邏輯，只需確保 Markdown 渲染器支援 Emoji 字元顯示。

---

### 3-3. 總結統計行格式規格

這是本次需求的核心輸出格式，附加於報告 Markdown 字串末尾：

**格式樣板：**

```
---
共發現 N 個 🔴 嚴重問題、M 個 🟡 建議
```

| 項目 | 規格 |
|------|------|
| 分隔線 | 總結行前必須插入一條 Markdown 水平線（`---`），與主體報告區隔 |
| N | 🔴 嚴重問題的計數，整數，最小值為 0 |
| M | 🟡 建議的計數，整數，最小值為 0 |
| Emoji | 🔴 對應嚴重問題、🟡 對應建議，字元直接嵌入字串 |
| 標點 | 中文頓號（、）分隔兩個統計項目，句尾無標點符號 |
| 換行 | 總結行前後各保留一個換行（`\n`），確保 Markdown 渲染正確斷行 |

**完整附加字串範例（字串層級）：**

```
\n---\n共發現 3 個 🔴 嚴重問題、5 個 🟡 建議
```

## 4. 資料來源

### 4-1. 統計數據來源

統計邏輯在**報告生成階段**執行，由 Vera 的報告生成服務負責，而非在 UI 層計算。

| 資料項目 | 來源 | 說明 |
|----------|------|------|
| 🔴 嚴重問題數量（N） | Vera 報告生成邏輯內部統計 | 掃描報告條目，計算標記為嚴重問題的項目數 |
| 🟡 建議數量（M） | Vera 報告生成邏輯內部統計 | 掃描報告條目，計算標記為建議的項目數 |
| 最終報告字串 | Vera 生成後寫入資料來源（DB / 快取） | UI 端直接取用已含總結行的完整字串 |

### 4-2. 統計識別規則（供 Dev Agent 確認）

> ⚠️ **此處需與 Dev Agent 確認現行報告格式**，以下為推測的識別規則，請依實際格式調整：

| 問題類型 | 識別依據（推測） |
|----------|-----------------|
| 🔴 嚴重問題 | 報告條目中包含 `🔴` 或對應的嚴重等級標記文字 |
| 🟡 建議 | 報告條目中包含 `🟡` 或對應的建議等級標記文字 |

**若目前報告條目使用的是文字標籤（如 `[嚴重]`、`[建議]`）而非 Emoji，請回報給 Demi 以更新識別規格。**

### 4-3. UI 層資料取用方式

| 項目 | 說明 |
|------|------|
| 取得方式 | 透過現有 API / SignalR（維持現行資料流，不新增端點） |
| 時機 | 報告生成完成事件觸發後，UI 訂閱更新並重新渲染報告字串 |
| 格式 | 報告字串本身已含總結行，UI 端無需額外處理 |

## 5. 互動行為

本功能以**靜態展示**為主，互動行為極簡：

### 5-1. 報告生成完成時

| 觸發點 | Vera 完成報告生成並寫入資料來源 |
|--------|-------------------------------|
| 系統行為 | UI 透過 SignalR 或 API polling 收到更新通知，重新渲染報告展示區 |
| 總結行展示 | 報告字串末尾的總結統計行隨主體內容一併渲染，無需額外觸發 |

### 5-2. 邊界情況行為

| 情況 | 期望行為 |
|------|---------|
| N = 0、M = 0 | 仍顯示總結行：「共發現 0 個 🔴 嚴重問題、0 個 🟡 建議」，不省略 |
| 報告主體為空 | 總結行仍附加（顯示 0/0），避免遺漏統計輸出 |
| 統計計算失敗 | 不附加總結行，由後端 log 錯誤，UI 僅顯示報告主體（降級處理） |

## 6. 注意事項

### 6-1. Blazor Server 特殊考量

| 項目 | 說明 |
|------|------|
| 電路隔離 | 報告字串由後端生成後存入 DB / 快取，不在 Circuit 狀態中暫存大型字串，避免記憶體壓力 |
| SignalR 更新 | Vera 生成完成後應透過 SignalR Hub 推送「報告已更新」事件，UI 接收後重新從 API 取得完整報告字串，而非直接傳輸整份報告字串於 SignalR 訊息中 |
| 渲染觸發 | 接收到 SignalR 更新後，需呼叫 `StateHasChanged()` 確保 UI 重新渲染 |

### 6-2. 統計邏輯位置

| 項目 | 說明 |
|------|------|
| 建議位置 | 統計與附加總結行的邏輯應集中在 Vera 的報告生成 Service 層，而非分散在 UI 元件或 Controller 層 |
| 不建議做法 | 在 UI 元件的 `OnInitializedAsync` 或 `OnParametersSetAsync` 中用字串 parse 報告內容來做統計，會增加 UI 複雜度且難以維護 |

### 6-3. Emoji 相容性

| 項目 | 說明 |
|------|------|
| 字元編碼 | 確認 DB 欄位（若有儲存報告）的字元編碼支援 UTF-8 4-byte Emoji（🔴、🟡 均為 Unicode 補充字元） |
| Markdown 渲染器 | 確認現行 Markdown 渲染元件不會 escape 或移除 Emoji 字元 |

### 6-4. 未來擴充考量（不在本次範圍，僅備註）

若未來需要新增問題類型（如 🔵 資訊、⚫ 阻斷）至統計，建議將總結行格式改為**可設定的模板**，而非 hardcode 字串拼接，以降低後續修改成本。本次實作無需處理此點。

在開始實作前，請回覆以下問題：

1. **現行報告條目的嚴重等級標記格式為何？** 是 Emoji（🔴/🟡）、文字標籤（`[嚴重]`/`[建議]`）、還是結構化物件？這影響統計邏輯的識別方式。
2. **報告字串是否有儲存至 DB？** 若有，確認欄位 collation / charset 支援 Emoji。
3. **現行 Markdown 渲染元件為何？** 確認其 Emoji 支援狀況。
